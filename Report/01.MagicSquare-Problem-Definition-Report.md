## Executive Summary

본 프로젝트의 진짜 과제는 「마방진 그림을 만드는 것」이 아니라, **4×4에 1~16을 배치했을 때 다중 제약(합·중복·범위)을 명시적 규칙으로 일관되게 판정·반복·고정하는 것**이다. 프로그램·TDD는 손검산의 변동을 줄이고, 완성(전 제약 동시 만족)의 기준이 흔들리지 않게 하기 위한 수단이다.

## STEP 1 — Observation (관찰)

### 1. 현재 우리가 해결하려는 상황은 무엇인가?

**한 사람이 4×4 격자에 1부터 16까지의 정수를 배치했을 때, 행·열·대각선 합이 모두 같은지 확인·탐색·생성하는 과정을 컴퓨터로 자동화하려는 상황**이다.

- **입력:** 4×4 격자(또는 1~16 숫자 집합), 때로는 빈 칸 또는 일부만 채워진 상태
- **규칙:** 각 행, 각 열, 두 대각선의 합이 동일 (4×4 표준에서는 34)
- **출력 기대:** 규칙을 만족하는 배치, 만족 여부 판정, 또는 가능한 배치를 찾아 보여 주는 결과
- **맥락:** 사람이 손으로 채우거나 검산하는 대신, **반복·조합·검증**을 프로그램에 맡기려는 의도

**관찰 표현:** 「16개의 서로 다른 정수를 4×4에 놓았을 때, 여러 방향의 합이 동시에 맞는 배치를 다루는 상황」

### 2. 왜 4×4 마방진 문제를 다루는가?

| 관점 | 관찰 |
|------|------|
| 규모 | 3×3보다 제약이 많고, 5×5보다 다루기 쉬움 — 한 프로젝트에 넣기 좋은 크기 |
| 수학 | 짝수 차(4×4)는 홀수 차와 생성·증명 방식이 다름 — 별도 문제로 인식됨 |
| 학습 | 배열, 순열, 불변식(합=34), 대칭·회전 등 기초 개념을 한 주제에 묶기 적합 |
| 실용(약함) | 퍼즐·교육용 — 규칙 검증 + 탐색 패턴 연습 |

4×4는 **「너무 작지도 않고, 손으로도 막히고, 자동화로는 여전히 의미 있는」** 경계 구간이다.

### 3. 학습·시스템 설계 맥락

**학습:** 2차원 구조, 다중 제약 검증, 불변식, (선택) 탐색·역할 분리(판정기/생성기/솔버)

**시스템 설계 (관찰 수준):**

```
[사용자/입력] → [배치 또는 부분 입력] → [검증·탐색·생성] → [결과·설명]
```

- 검증 / 생성 / 솔버 / 동치(회전·대칭) 표현 — 서브문제로 갈라짐
- 프로젝트명 `MagicSquare_`는 도메인 중심 단일 목적 애플리케이션 신호

### 관찰 요약

「4×4에 1~16을 배치할 때, 여러 방향의 합이라는 동시 제약을 만족하는 상태를 프로그램이 다루게 하려는 상황」이며, 학습·연습에 적당한 난이도와 짝수 차 마방진 이슈가 겹친다.

### 열린 질문 (다음 단계용)

- 규칙 **검증**만?
- **하나 생성**?
- **모든 해** 또는 **대표 해**?
- **빈 칸 퍼즐** 풀이?

## STEP 2 — Why #1

### Q: 왜 마방진을 완성해야 하는가?

**가정한 답 (A):**  
4×4 격자에 1~16을 놓았을 때, 행·열·대각선 합이 모두 같다는 조건을 만족하는 **완성 상태**를 얻어야, 그 배치가 올바른지 판단하거나 퍼즐·과제·시험 요구를 충족할 수 있다. 완성 = 규칙을 모두 만족하는 종착 상태. 그 전에는 맞다/틀리다를 단정하기 어렵다.

### 불편함

1. **「완성」과 「맞음」이 묶임** — 중간 과정(일부만 채운 판)도 의미 있으나, 16칸 종착만 인정될 수 있음. 한 칸만 틀려도 vs 합만 우연히 맞는 가짜 완성 구분 어려움.
2. **완성 목표가 하나로 고정되지 않음**

| 목적 | 「완성」의 의미 |
|------|----------------|
| 학습·증명 | 아무 유효 배치 하나 |
| 퍼즐 | 힌트와 맞는 그 해 하나 |
| 탐구 | 서로 다른 해 전부 또는 대표 집합 |
| 발표 | 보기 좋은 / 대칭적인 한 배치 |

3. **사람의 단계적 검증 vs 전역 검증** — 행 합 → 열 → 대각 순 vs 전 제약 동시 만족.
4. **4×4 난이도** — 탐색 어려움 vs 규칙 기계 확인. Why 목적에 따라 설계가 달라짐.

### 구조적 문제

| 구조 | 내용 |
|------|------|
| 종착 vs 과정 | 완성만 강조하면 중간 상태·진단 개념이 빈약 |
| 단일 정답 환상 | 회전·대칭·동치류 미정의 |
| 제약 계층 숨김 | 순열, 행·열·대각 합 — 우선순위·필수 순서 없음 |
| 책임 주체 | 사용자 완성 vs 프로그램 완성 vs 협업 — 미분리 |

### Why #1 정리

| 항목 | 내용 |
|------|------|
| 가정한 이유 | 전 제약 만족 종착 상태만이 올바름·과제 충족 보장 |
| 핵심 불편 | 완성 전 피드백 부재, 완성 목적·동치 혼동, 단계 vs 전역 검증 불일치 |
| 구조 문제 | 종착 중심 정의, 단일 정답 가정, 제약·주체 미분리 |

## STEP 3 — Why #2

### Q: 왜 단순 계산이 아니라 프로그램으로 구현하는가?

### 1. 반복 가능성 (Repeatability)

**답:** 같은 규칙·입력에 대해 **동일 절차와 결과** — 손검산은 변동이 크고, 프로그램은 절차 고정·대량 시도·실험 재현에 유리.

| 손으로 할 때 | 프로그램으로 할 때 |
|--------------|----------------------|
| 피로·순서 바뀜·누락 | 검증·생성·출력 순서 고정 |
| 「또 하나」 탐색 부담 | 수천·수만 번 시도·나열 가능 |
| 실험마다 기록 새로 작성 | 입력만 바꿔 동일 파이프라인 재실행 |

**이슈:** 반복 단위(한 해 / 모든 해 / 매 시도) 정의 필요.

### 2. 검증 자동화 (Verification Automation)

**답:** 행·열·대각·중복·범위를 **매번 전부** 기계 판정 — 사람은 일부만 보고 착각하기 쉬움.

- 판정: 합격/불합격 즉시
- 진단(선택): 어떤 제약이 깨졌는지
- 채점·튜터: 사람 개입 없이 동일 기준

**이슈:** 「맞는지」≠ 「맞는 것을 찾기」; 동치 비교 규칙 추가 필요.

### 3. 오류 방지 (Error Prevention)

**답:** 합산·중복·절차 누락을 기계에 이양.

| 오류 유형 | 손작업 | 프로그램 |
|-----------|--------|------------|
| 산술 | 계산 착오 | 합을 규칙으로 계산 |
| 논리 | 중복·누락 | 집합·개수 검사 |
| 절차 | 대각선 미검 | 검증 목록 빠짐없이 실행 |

**이슈:** 프로그램 버그·잘못된 규칙 구현; 입력 실수는 별도 통제.

### 4. 규칙 기반 사고 훈련 (Rule-Based Thinking)

**답:** 암묵적 감각 → **상태·규칙·전이·종료** 명시 — 규칙을 검증 가능하게 쓰는 연습.

**이슈:** 최소 검증기 vs 풀 솔버; 4×4 전용 트릭 vs 일반화.

### Why #2 통합 (가정 A)

| 관점 | 요약 |
|------|------|
| 반복 가능성 | 동일 절차로 많은 판·시도·실험 |
| 검증 자동화 | 다중 제약 빠짐없이·일관 판정 |
| 오류 방지 | 인지 부담 기계 이양 |
| 규칙 기반 사고 | 명시 규칙·상태·전이 |

**통합:** 4×4는 **반복 검사·다양한 입력·명시적 규칙**이 핵심이며, 프로그램은 절차·판정·표현으로 손검산 한계를 메운다.

**열린 선택:** 검증만 vs 생성·탐색; 교육 vs 도구 1순위.

## STEP 4 — Why #3 (TDD)

### Q: 왜 TDD 방식으로 설계하려 하는가?

**가정 A:** 제약이 많아 「그럴듯하지만 틀린」 결과가 나오기 쉽다. TDD는 제약을 테스트로 고정해, 기능 추가(검증 → 생성 → 힌트) 순서가 바뀌어도 회귀 없이 규칙을 지키게 한다.

### 무엇이 통제되어야 하는가?

**도메인**

| 통제 영역 | 내용 |
|-----------|------|
| 판정 기준 | 합격/불합격 계약 |
| 제약 순서 | 행만 통과 등 누락 방지 |
| 경계 | 4×4, 1~16 |
| 동치 | 회전·대칭 정책 명시 |
| 범위 확장 | 검증 → 생성 폭주 완화 |
| 회귀 | 리팩터 후 규칙 유지 |

**프로세스:** Why #2의 반복·자동화를 개발 단계로 이동 — CI/로컬 동일 테스트, RED→GREEN→REFACTOR. 검증을 층(행/열/대각/중복)으로 쪼개 통제.

### 불변 조건

**도메인 불변**

- 격자 4×4; 값 1~16; 완성 시 1~16 각 1회
- 행·열·주대각·부대각 합 = 34
- 전 제약 만족 ⇔ 유효

**탐색·부분판 (확장 시)**

- 이미 채운 칸 불변; 깨진 제약은 더 깊이 가지 않음

**정책 불변 (정해야 함)**

- 입출력 계약; 동치(엄격 일치 vs 대칭·회전 후 동일)

### 왜 입출력이 명확해야 하는가?

테스트 = **같은 입력 → 같은 기대 출력**.

- 빈 칸 / 크기 / 부분판 → 검증 vs 솔버 혼합
- true/false만 → 진단·과정 피드백 불가
- 해 출력 → 동치·재현성 애매

**TDD가 요구하는 것:** (1) 테스트가 문서 (2) 실패 메시지 (3) 단계적 구현 (4) 반례로 가짜 통과 방지

**열린 결정:** 검증만 vs 생성 포함; boolean vs 위반 목록; 엄격 배열 vs 동치 정규화.

## STEP 5 — 진짜 문제 정의

**제약:** 구현 설계 없음 · 코드 없음 · 알고리즘 설명 없음

### 1. 표면 문제 정의 (잘못된 정의)

> **4×4 격자에 1부터 16까지 숫자를 넣어, 행·열·대각선의 합이 모두 같은 마방진을 만드는 프로그램을 만든다.**

| 함정 | 설명 |
|------|------|
| 행위만 강조 | 생성만 — 판정·진단·부분·반복 누락 |
| 목표 하나처럼 보임 | 하나의 해 / 모든 해 / 힌트 해 불명 |
| 완성만 성공 | 학습·검증 과정 사라짐 |
| 도구 수준 미정 | 왜 프로그램·반복·규칙 고정인지 없음 |
| 정답 환상 | 동치·회전·대칭 관계 없음 |
| TDD·품질 누락 | 통제·불변 없이 기능 목록 |

### 2. 개선된 문제 정의 (정확한 정의)

> **4×4에 1~16을 각각 한 번씩 배치했을 때, 행·열·두 대각선의 합이 모두 동일한지(및 그에 필요한 모든 조건을) 사람의 반복 검산에 의존하지 않고, 명시된 규칙으로 일관되게 판단·기록·반복할 수 있게 한다. 필요 시, 규칙을 만족하는 배치를 찾거나, 주어진 부분 배치에 맞는 완성 상태까지 다룬다. 개발과 운용 전 과정에서 「무엇이 맞는 판인가」의 기준이 흔들리지 않도록, 그 기준을 검증 가능한 형태로 고정한다.**

| 요소 | 내용 |
|------|------|
| 대상 | 4×4, 1~16 배치(완성·부분) |
| 핵심 과제 | 다중 제약 빠짐없이 적용·동일 기준 반복 |
| 프로그램 역할 | 변동·누락·피로 감소, 규칙 명시 |
| 종착 상태 | 전 제약 동시 만족 (유일 그림 아님) |
| 품질 | 기준 변경 시에도 이전 판정 유지 |
| 범위 | 1차 판정(·진단) 중심; 생성·퍼즐은 확장 |

**성공 기준**

- 맞음/틀림이 항상 같고 설명 가능
- (선택) 위반 종류 구분
- 동일 입력 → 동일 결론
- 「완성」= 전 제약 동시 만족

### 3. 핵심 Invariant

| ID | 불변 | 설명 |
|----|------|------|
| I1 | 격자 크기 | 항상 4행×4열 |
| I2 | 값 범위 | 각 칸 ∈ [1, 16] |
| I3 | 완성 시 순열 | 1~16 각 정확히 1회 |
| I4 | 마법 상수 | 공통 합 = 34 |
| I5 | 행 합 | ∀행 합 = 34 |
| I6 | 열 합 | ∀열 합 = 34 |
| I7 | 대각 합 | 주·부대각 합 = 34 |
| I8 | 판정 일관성 | I3~I7 모두 만족 시에만 유효 완성 |
| I9 | 부분 위반 전파 | 부분에서 이미 깨지면 유효 완성 불가 |
| I10 | 재현성 | 동일 입력 → 동일 판정 |
| I11 | 입력 계약 | 완성/부분/거부 정의 |
| I12 | 출력 계약 | 판정·진단 의미 고정 |
| I13 | 동치 정책 | 같은 해 정의 명시 |

### 4. 훈련하려는 사고 능력
