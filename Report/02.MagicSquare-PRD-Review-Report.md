## Executive Summary

| 기준 | 판정 | 요약 |
|------|------|------|
| 1. FR별 테스트 가능 AC | **부분 충족** | 5개 FR 모두 AC 존재. 다만 성공 경로·골든 출력·두 번째 빈칸 정의 AC가 약함. |
| 2. AC ↔ Traceability Matrix | **미충족** | Matrix는 Concept 중심이며 AC ID 전수 매핑 없음. 5개 이상 AC 누락. |
| 3. Boundary / Domain 책임 분리 | **부분 충족** | FR-05·BR-02·출력 검증 책임에서 계층 혼재 흔적 있음. |

| 5. small-first / reverse 데이터 구분 | **부분 충족** | TD-01/TD-02 분리는 됨. 기대 출력·Attempt1 실패 증명 요건 없음. |
| 6. 1-index / row-major 규칙 | **부분 충족** | 첫 빈칸·1-index는 있음. **두 번째 빈칸** row-major 정의 누락. |

**권고:** Traceability Matrix를 AC ID 단위로 재작성하고, FR-05를 Control(시도)과 Boundary(포맷)로 분리하며, TD-01/TD-02에 기대 `int[6]` 및 reverse 전제 조건을 AC에 고정할 것.

## 1. 모든 FR에 테스트 가능한 AC가 있는가?

**판정: 부분 충족**

### FR별 AC 현황

| FR | AC ID | 테스트 가능성 | 문제 |
|----|-------|---------------|------|
| FR-01 | AC-FR01-01 ~ 05 | 대체로 가능 | 성공(검증 통과) AC 없음. `Domain resolver` 용어 미정의. |
| FR-02 | AC-FR02-01, 02 | 부분 가능 | row-major **첫 번째**만 암시. **두 번째 빈칸** 순서 AC 없음. 골든 좌표 없음. |
| FR-03 | AC-FR03-01, 02 | 가능 | 입력 행렬 예시·기대 `[small,large]` 쌍 미고정. |
| FR-04 | AC-FR04-01 ~ 04 | 가능 | 부분 채움/가상 채움 후 판정 절차 AC 없음. |
| FR-05 | AC-FR05-01 ~ 05 | 부분 가능 | 성공 시 기대 `int[6]` 값 AC 없음. 출력 길이/1-index는 BS에만 분산. |

### 누락·모호 AC 목록

| ID | 문제 유형 | 설명 | 개선안 |
|----|-----------|------|--------|
| (없음) | 누락 | FR-01 검증 **성공** 시 다음 단계 진입 AC 없음 | `AC-FR01-06`: 유효 입력이면 예외 없이 Control에 위임한다. |
| AC-FR02-01 | 모호 | “row-major 순서로 반환”만 있고 좌표 값 검증 불가 | TD-01 기준 `AC-FR02-03`: 첫 빈칸=(1,2), 둘째=(1,3) 등 **고정 기대값** 추가. |
| (없음) | 누락 | 두 번째 빈칸 = row-major 상 두 번째 `0` | `AC-FR02-04` + `BR-05b` 신설. |
| AC-FR01-05 | 모호 | “Domain resolver” 컴포넌트/함수 경계 미정 | `Solver` 또는 `Control.resolve` 호출 금지를 mock/spy로 AC에 명시. |
| AC-FR05-03 | 모호 | “그대로 반환”은 Attempt2 성공 시 `n1>n2` 가능 — 의도 확인 필요 | reverse 성공 시 `n1=large, n2=small`인지 **출력 순서 계약**을 한 문장으로 고정. |
| (없음) | 누락 | `ResultFormatter` / 출력 계약 전용 AC | `AC-FR05-06`: 반환 배열 길이=6, 모든 좌표∈[1,4]. |

### 모순 가능성

- **출력 `n1,n2` 의미:** 사용자 고정 계약은 “누락된 두 숫자”이나, FR-05 AC-FR05-03은 “시도 순서 반영”으로 읽힌다. reverse 성공 시 출력이 `[r1,c1,large,r2,c2,small]`인지 `[r1,c1,small,r2,c2,large]`인지 PRD만으로 단정 불가 → **Decision Needed** 수준의 모호함(본문에는 예외로 분리되지 않음).

## 2. 모든 AC가 Traceability Matrix에 연결되어 있는가?

**판정: 미충족**

### Matrix에 **없는** AC ID

| AC ID | FR | 비고 |
|-------|-----|------|
| AC-FR01-05 | FR-01 | Domain 미호출 — Track A 핵심 AC |
| AC-FR02-02 | FR-02 | 빈칸 2개 — Concept “빈칸 2개”와만 간접 연결 |
| AC-FR03-01 | FR-03 | 누락 개수 2 — Matrix에 AC-FR03-02만 명시 |
| AC-FR04-01 | FR-04 | 행 합 — `AC-FR04-01~03`으로 묶여 개별 추적 불가 |
| AC-FR04-02 | FR-04 | 열 합 | 동일 |
| AC-FR04-03 | FR-04 | 대각 합 | 동일 |
| AC-FR05-03 | FR-05 | 매핑 그대로 반환 | Matrix는 AC-FR05-01/02/04만 연결 |
| AC-FR05-05 | FR-05 | `E-DOM-001` | Error §13에만 있고 Matrix 행 없음 |

### Matrix 구조적 한계

- Matrix가 **Concept → AC**가 아니라 **Concept → 일부 AC** 매핑이라 “모든 AC 연결” 요건을 충족하지 못함.
- `Test Case Candidate`가 `ES-01`, `Track B blank order` 등 **ID 체계 불일치**(§16 NS/ES/BS/TD vs §23 TA/TB-RED).
- Gherkin G-01~G-04가 Matrix에 없음.

### 개선안

1. Matrix 1행 = 1 AC ID 형식으로 확장.
2. 열 추가: `Error Code`, `Test Data ID`, `Gherkin ID`.
3. §16 `TD-xx` / §23 `TB-RED-xxx` / Matrix `Test Case Candidate` **단일 ID 체계**로 통합.

## 3. Boundary와 Domain 책임이 섞인 곳은 없는가?

**판정: 부분 충족 (혼재 지점 존재)**

### 혼재·경계 불명확 지점

| 위치 | 문제 | 영향 | 개선안 |
|------|------|------|--------|
| FR-05 Layer | `Control + Boundary(Output format)` 단일 FR | Track A/B 경계 흐림 | FR-05a Solver(Control), FR-05b Output Formatting(Boundary) 분리 |
| §9.1 In-Scope | “출력 계약 검증” 전용 FR 없음 | FR-05에 흡수되어 Boundary 책임 과다 | FR-06 Output Contract Validation (Boundary) 신설 또는 FR-01 확장 |
| BR-02 | Boundary 검증(FR-01) + Domain 빈칸 개수(FR-02) 공유 | 동일 규칙 이중 소유 | BR-02를 Boundary 전용으로 두고 Domain은 “FR-01 통과 입력만 수신” 전제로 중복 제거 |
| §12.1 First Blank | 실패 시 “도메인 판정 불일치” | Boundary vs Domain 오류 소속 불명 | 입력 계약 위반이면 `E-BND-*` 또는 Solver 입력 오류로 귀속 명시 |
| §17 Boundary | “성공 시 출력 포맷 보장” | 포맷 검증 vs 생성 혼동 | Boundary=검증+변환, Control=조합 결정, Domain=불변식만 |
| FR-02 Output | “내부 0-index 허용” | 1-index는 Boundary 책임인데 FR-02 Domain에 혼재 문구 | Domain은 0-index 좌표 반환, Boundary만 1-index 변환 — AC를 계층별로 분리 |
| Component `Solver` | Layer=Control, Related Test=BS-04/05 | BS는 Boundary 시나리오 | BS-04/05를 `ResultFormatter` 또는 FR-05b에만 연결 |

### 잘 분리된 부분

- FR-01 = Boundary 전담, 실패 시 Domain 미호출(AC-FR01-05).
- FR-04 = Domain 판정 전담.
- 의존 방향 `Boundary → Control → Domain` 명시.

## 4. 오류 정책이 미정으로 남아 있지 않은가?

**판정: 부분 충족**

### 확정된 정책 (양호)

| 실패 유형 | Error Code | Domain 호출 | 관련 AC |
|-----------|------------|-------------|---------|
| 4x4 아님 | E-BND-001 | 금지 | AC-FR01-01, 05 |
| 빈칸 개수 ≠ 2 | E-BND-002 | 금지 | AC-FR01-02, 05 |
| 값 범위 위반 | E-BND-003 | 금지 | AC-FR01-03, 05 |
| 0 제외 중복 | E-BND-004 | 금지 | AC-FR01-04, 05 |
| 두 조합 실패 | E-DOM-001 | 허용(검증 후) | AC-FR05-05 |

§13 “실패 시 예외, 성공 시만 `int[6]`” — **미정 아님**.

### 미정·모호·모순

| 항목 | 상태 | 설명 | 개선안 |
|------|------|------|--------|
| 예외 **타입** | 미정 | `Boundary Validation Exception` / `Domain Resolution Exception`의 클래스·상속·코드 필드 규약 없음 | §13에 `exception_type`, `error_code` 속성 AC 추가 |
| 출력 계약 위반 | 미정 | §12.2 “계약 위반(테스트 실패)” — Error Code 없음 | `E-BND-005` 또는 내부 assert 정책 중 택1 |
| Message 언어 | Decision Needed 03 | 본문은 영문 Message, 팀 언어 미확정 | §22 결론 반영 전까지 AC는 `error_code`만 검증 |
| Coverage 80% vs 95/85 | Decision Needed 04 | NFR과 Cursor Rules 충돌 | §14 단일 수치 확정 |
| `int` 외 입력 | 미정 | §12 Invalid `1.5` 예시 vs `int[][]` 타입 | Python에서는 int만 — `float`/문자열 거부 AC 추가 여부 결정 |
| TD-07 검증 | 미검증 명시 | two-attempt failure 데이터만 있고 `E-DOM-001` 기대 연결 약함 | `AC-FR05-05` + TD-07 + TB-RED-009 삼각 연결 |

## 5. small-first 성공과 reverse 성공 테스트 데이터가 구분되어 있는가?

**판정: 부분 충족**

### 충족

- §16.4 **TD-01** (small-first 성공), **TD-02** (reverse 성공) — 행렬 **분리**됨.
- §16.1 NS-01 / NS-02 시나리오 이름 분리.

### 미흡·모호

| 항목 | 문제 | 개선안 |
|------|------|--------|
| TD-01 / TD-02 | **기대 `int[6]` 출력값 없음** | TD-01: `AC-OUT-01` 기대 배열 명시. TD-02: `AC-OUT-02` 명시. |
| TD-02 | “reverse 성공” 라벨만 있고 **Attempt1 실패 전제 미검증** | `AC-FR05-07`: TD-02에서 Attempt1 결과는 마방진이 아니다. |
| NS-01/02 ↔ TD | 참조 링크 없음 | NS-01=TD-01, NS-02=TD-02로 ID 교차 참조 |
| TB-RED-007/008 | §23 후보만 있고 TD 미연결 | TB-RED-007=TD-01, TB-RED-008=TD-02 고정 |
| 데이터 검증 | TD-01/02가 실제로 각 시나리오를 만족하는지 PRD 내 증명 없음 | Appendix에 “검증 완료” 체크리스트 또는 수동 검산 표 추가 |

### 모순 위험

- TD-01과 TD-02가 **동일한 빈칸 위치 패턴**이 아님 → 좋음.  
- 다만 리뷰어가 Attempt 순서만 바꿔 동일 행렬로 오해할 수 있음 → §19 Risk에 대응책은 있으나 **AC로 강제되지는 않음**.

## 6. 1-index 좌표와 row-major 규칙이 누락되지 않았는가?

**판정: 부분 충족**

### 포함된 규칙

- BR-05: row-major **첫 번째** 빈칸.
- BR-12, AC-FR05-04, BS-04: 1-index 출력.
- §12.2 Coordinates 1~4.

### 누락·모호

| 규칙 | 상태 | 개선안 |
|------|------|--------|
| **두 번째 빈칸** row-major 정의 | **누락** | BR-05b: 첫 `0` 이후 스캔에서 다음 `0`. AC-FR02-04 추가. |
| 빈칸 좌표 → 출력 좌표 매핑 | 모호 | FR-02는 0-index, 출력은 1-index — `AC-FR05-08`: (r0,c0)→(r0+1,c0+1) |
| row-major 스캔 시작점 | 암묵 | BR-05에 “(0,0)부터 행 우선” 명시 |
| 대각선 정의 | 암묵 | BR-09에 주대각/부대각 인덱스 정의(예: (0,0)-(3,3), (0,3)-(3,0)) |
| Traceability | 부분 | Matrix에 “두 번째 빈칸” Concept 행 없음 |

## 7. 구현 코드나 테스트 코드가 포함되어 있지 않은가?

**판정: 충족**

- Python 소스, pytest 테스트 함수, `@pytest` fixture 코드 **없음**.
- `int[][]`, `int[6]` — 계약 표기로 적절.
- 컴포넌트 후보명(`BoundaryValidator` 등) — 설계 후보 수준.
- §23 Gherkin — 시나리오 명세이지 실행 코드 아님.

### 주의 (오판 방지)

- §16.4 행렬 리터럴은 **테스트 데이터 명세**이지 테스트 코드가 아님 → 유지 가능.
- 향후 `docs/PRD_MagicSquare.md` 저장 시 구현/테스트 파일과 혼재하지 않도록 저장 경로 분리 권장.
